# /hk:pre-clear:save

`/clear`로 세션을 비우기 직전, **다음 세션이 흐름을 이어받을 수 있도록** 현재까지의 진행 상태와 앞으로 할 일을 핸드오프 파일로 남긴다.

- 저장 위치: `.hk/pre-clear/handoff.md` (gittracked)
- 짝 명령: `/hk:pre-clear:resume` — 새 세션에서 이 파일을 읽고 자동 삭제
- Memory가 아닌 .md 파일에 저장하는 이유: 진행 중 작업은 "ephemeral task details"라 memory 정책상 저장 금지 + MEMORY.md 200줄 truncate 제약

## 실행 절차

### 1) 기존 핸드오프 파일 정리

```bash
rm -f .hk/pre-clear/handoff.md
mkdir -p .hk/pre-clear
```

기존 파일이 있으면 **확인 없이 덮어쓴다** (사용자가 명시적으로 동의한 정책 — 1회성 파일).

### 2) 현재 세션 분석

다음 항목을 **현재 컨텍스트에서 직접** 추려낸다 (별도 도구 호출 없이):

| 항목                            | 내용                                                                              |
| ------------------------------- | --------------------------------------------------------------------------------- |
| **세션 목표**                   | 사용자가 이번 세션에서 하려던 일 1~2 문장                                         |
| **완료한 작업**                 | 실제로 끝난 것만. 추정/계획 단계는 제외. 파일 변경은 `path:line` 형식으로         |
| **진행 중 작업**                | 어디까지 갔고 무엇이 남았는지. 다음 단계가 뚜렷해야 함                            |
| **다음에 할 일**                | 새 세션이 우선 손댈 항목 (1~5개, 우선순위 순)                                     |
| **결정된 사항**                 | 사용자와 확정한 정책·선택 (재논의 방지). 이번 국면 한정 도구/역할 분담은 아래 작업 메모리로 |
| **작업 메모리**                 | 워크플로우·역할 분담·도구/모델 선택 등 이어서 기억할 운영 컨텍스트 (예: 코드=Codex, 리뷰·검증=Claude). 결정된 사항과 겹치면 "작업 끝나면 무의미해지는가"로 가른다 — MEMORY.md엔 안 올리는 현재 국면 한정 |
| **미해결 질문**                 | 사용자 결재 대기 중이거나 보류된 항목                                             |
| **참조 파일**                   | 다음 작업 1순위를 시작할 때 즉시 열게 될 파일만. 부수 참고는 빼고 작성자가 추린다 |
| **Suggested Skills / Commands** | 다음 세션에서 바로 쓰면 좋은 skill/command 1~3개                                  |
| **주의 사항**                   | 함정·전제·환경 의존성 (있을 때만)                                                 |

### 3) handoff.md 작성

먼저 stale 감지·복원 앵커 값을 캡처한다 (git 프로젝트가 아니면 해당 값 생략):

```bash
date -u +%Y-%m-%dT%H:%M:%SZ            # 저장 시각
git rev-parse --short HEAD 2>/dev/null # HEAD SHA
git branch --show-current 2>/dev/null  # 브랜치/워크트리
```

아래 템플릿으로 `.hk/pre-clear/handoff.md` 작성 (메타 라인에 위 값 채움):

```markdown
# Session Handoff

> 다음 세션 시작 시 `/hk:pre-clear:resume` 호출.
> 저장: <ISO8601> · HEAD: <short-sha> · 브랜치: <name>

## 세션 목표

<1~2 문장>

## 완료한 작업

- <항목> — `path:line`
- ...

## 진행 중 작업

<어디까지 갔는지 + 무엇이 남았는지>

## 다음에 할 일

1. <우선 작업>
2. ...

## 결정된 사항

- <결정> — 이유
- ...

## 작업 메모리 (있으면 — 워크플로우·역할·임시 컨텍스트)

> 작업이 어떻게 굴러가고 있는지 — 다음 세션이 모르면 흐름이 끊기는 운영 컨텍스트. MEMORY.md에 올리기엔 영속적이지 않지만 `/clear`를 넘겨 이어가야 하는 것.

- <역할/도구 분담> — 예: 코드 구현 = Codex 5.6 sol, 플래닝·리뷰·검증 = Claude
- <이번 작업에서 합의된 진행 규칙·컨벤션> — 예: 커밋 단위, 브랜치/워크트리, 특정 도구 사용 규칙
- <활성 환경/모델 세팅> — 예: 어떤 워크트리에서 무슨 모델로, 어떤 MCP를 쓰는 중
- <MEMORY.md엔 안 올리지만 다음 세션이 모르면 흐름이 끊기는 임시 사실>

## 미해결 질문 (있으면)

- <질문> — 누구 결재 대기

## 참조 파일 (다음 작업 1순위에 즉시 필요한 것만)

- `path` — 역할
- ...

## Suggested Skills / Commands (있으면)

- `<skill-or-command>` — <왜 필요한지>

## 주의 사항 (있으면)

- <함정·전제>
```

### 4) 기록 검증 (게이트 — 통과해야 `/clear` 승인)

이 커맨드의 존재 이유는 `/clear`로 컨텍스트를 잃지 않는 것이다. 파일이 실제로·완전히 기록됐는지 확인한다:

```bash
test -s .hk/pre-clear/handoff.md \
  && grep -q '^# Session Handoff' .hk/pre-clear/handoff.md \
  && grep -q '^## 세션 목표' .hk/pre-clear/handoff.md \
  && grep -q '^## 다음에 할 일' .hk/pre-clear/handoff.md \
  && echo OK || echo FAIL
```

**`OK`일 때만 5)로 진행**한다. `FAIL`이면(빈 파일·잘못된 cwd·권한·부분 기록) 사용자에게 사실을 보고하고 **절대 `/clear`를 권하지 않으며** 재작성한다.

### 5) 저장 후 보고

사용자에게 다음을 한 번에 보고:

- 저장 경로 (`.hk/pre-clear/handoff.md`) + 검증 결과(OK)
- 항목별 라인 수 또는 요약
- 다음 세션에서 호출할 명령 안내: `/hk:pre-clear:resume`

### 6) 마무리 안내

> (검증 OK일 때만) 이제 `/clear`로 세션을 비워도 됩니다. 새 세션 시작 후 `/hk:pre-clear:resume`을 호출하면 컨텍스트가 복원되고 핸드오프 파일은 자동 삭제됩니다.

## 작성 원칙

- **사실만 기재**. 추측·계획 단계 배제.
- **path:line 인용** 가능하면 항상.
- **완전성 우선 (간결함보다)** — 포함 여부가 애매하면(관련성이 borderline) 담는다. 핸드오프 몇 줄 더 쓰는 비용은, 다음 세션이 잃어버린 맥락을 다시 캐내거나(못 캐서 흐름이 끊기는) 비용보다 훨씬 싸다. 압축하다 신호를 잃지 말 것 — 토큰보다 연속성이 비싸다. (단 ① raw 대화 덤프가 아니라 구조화된 섹션에 넉넉히, ② 사실이 불확실한 건 추측이므로 담지 말고 「미해결 질문」으로 옮긴다.)
- **결정 이유 한 줄** — 새 세션이 같은 토론을 반복하지 않게.
- **3계층 분리** — 영속 프로필·정책은 MEMORY.md, 현재 국면의 워크플로우·역할·임시 운영 컨텍스트는 **작업 메모리** 섹션, 순수 진행 상태는 나머지 섹션. 작업 메모리엔 "이번 작업이 끝나면 무의미해질" 운영 사실만 담는다 (모델/도구 분담·임시 컨벤션·활성 환경 등).
- **민감정보 redaction** — API key, token, password, 개인정보는 직접 쓰지 않는다. 필요하면
  git-tracked 문서 경로만 참조한다.
- **중복은 참조로, 단 사라질 것은 인라인** — PRD·plan·ADR·issue·commit·diff처럼 안정적으로
  남는 산출물은 복사하지 말고 path / commit / URL 로 참조한다. 하지만 참조 대상이 곧
  사라지거나(미커밋 상태·임시 브랜치) 다음 세션이 접근 못 할 수 있으면 핵심은 인라인으로
  담는다 — 접근 불가한 참조는 압축 손실과 같다.
- **Suggested Skills / Commands** — 다음 세션에서 바로 쓸 skill/command가 있으면 1~3개만 적는다.

## 안티패턴

| ❌                                     | 이유                                                                  |
| -------------------------------------- | --------------------------------------------------------------------- |
| 전체 대화 덤프                         | 노이즈 과다. 다음 세션이 추리는 데 더 오래 걸림                       |
| 코드 블록을 통째로 복사                | 파일에 이미 있는 내용. `path:line`로 인용                             |
| 추측을 기록                            | "아마 ~일 것" 같은 표현. 사실만                                       |
| 비밀번호/API key/token 직접 기재       | handoff는 1회성이라도 파일이다. 민감정보는 문서 경로/계정 위치만 참조 |
| smoke report/QA tracker 내용 장문 복붙 | 이미 산출물이 있으면 경로와 핵심 결론만 참조                          |
| 기존 handoff.md 보존                   | 1회성 정책 위반. 무조건 덮어쓰기                                      |
| memory에 저장 시도                     | 진행 중 작업은 memory 정책상 저장 금지                                |
